# 项目模版蓝图

## 目标

这个模版面向“基于 `@gaozh1024/rn-kit` 启动一个真实业务 App”的场景。

目标是四件事：

- Provider 初始化顺序固定，开箱可跑
- 目录层级稳定，不在同一类职责上出现两套放法
- 首版页面覆盖基础产品能力
- 结构不过度设计，后续还能自然扩展

## 推荐模版位置

如果后续要在仓库里正式加入项目模版，建议目录为：

```text
templates/expo-starter/
```

不要把模版直接塞进 `packages/framework`，也不要复用 demo app 作为模版。  
demo app 适合做能力验证，不适合做业务项目起点。

## 各目录职责

### `app/`

只保留 Expo 入口文件。  
不要把业务页面继续堆进 `app/`。

`app/_layout.tsx` 只负责：

- 导入 `react-native-reanimated`
- 导入 `global.css`
- 挂载 `RootApp`

### `src/app/`

这里只放应用级装配代码。

- `RootApp.tsx`
  负责应用启动流程，决定先进入哪个导航分支
- `providers.tsx`
  统一包裹 `AppProvider`，挂主题和全局配置

### `src/bootstrap/`

这是启动配置层，放“应用启动时就需要知道的固定配置”。

- `app-config.ts`
  放应用配置，例如 `appName`、`env`、`apiBaseURL`、功能开关
- `theme.ts`
  放亮色/暗色主题配置
- `constants.ts`
  放启动阶段常量，例如默认语言、默认主题模式、缓存 key

`apiBaseURL` 就应该放在这里，不应该散落到别的目录。

### `src/navigation/`

这里不是重复造导航框架，而是定义“当前项目自己的路由结构”。

框架提供的是：

- `NavigationProvider`
- `StackNavigator`
- `TabNavigator`
- `DrawerNavigator`

项目自己仍然必须定义：

- 有哪些页面
- 哪些页面属于 `Auth`
- 哪些页面属于 `MainTabs`
- 页面之间怎么跳转
- route params 是什么

所以 `navigation/` 是必须保留的。

### `src/features/`

这里是按业务域组织的主目录。  
之所以不用全局 `screens/` 扁平铺开，是因为一旦出现“auth 内公共组件”“profile 内公共组件”“每个业务域各自有 API”，扁平结构会很快失控。

这里每个业务域只保留 3 层以内：

- `screens/`
- `components/`
- `api.ts`

例如：

- `features/auth/components/`
  放 `auth` 域内复用组件
- `features/profile/components/`
  放 `profile` 域内复用组件
- `features/profile/api.ts`
  放 `我的 / 用户信息 / 设置 / 主题 / 多语言 / 关于我们` 相关接口

### `src/components/`

这里只放跨业务域复用的公共组件。

- `common/`
  放通用组合组件，例如空态块、分组标题、表单区块、版本信息块

规则很简单：

- 只在 `auth` 里复用，就放 `features/auth/components`
- 只在 `profile` 里复用，就放 `features/profile/components`
- 跨多个业务域复用，才放 `components/common`

### `src/data/`

这是全局数据基础层，不承载所有业务域的具体接口。

建议分工：

- `api.ts`
  读取 `bootstrap/app-config.ts` 中的 `apiBaseURL`，提供统一的 API 创建入口和全局错误监听
- `schemas.ts`
  放跨业务域复用的 schema
- `session.ts`
  放 token、登录态、本地会话读写
- `mocks/*`
  放全局初始化数据

业务域自己的 API 放到各自目录：

- `features/auth/api.ts`
- `features/home/api.ts`
- `features/profile/api.ts`

### `src/store/`

首版只保留两份全局状态：

- `session.store.ts`
  登录态、当前用户、token
- `ui.store.ts`
  主题模式、当前语言、局部 UI 标记

## 启动链路

推荐启动顺序：

```text
app/_layout.tsx
  -> src/app/RootApp.tsx
  -> src/app/providers.tsx
  -> AppProvider
  -> RootNavigator
```

其中：

- `app/_layout.tsx` 只做入口挂载
- `providers.tsx` 统一注入 `AppProvider`
- `RootApp.tsx` 负责启动期判断
- `RootNavigator.tsx` 负责路由装配

## 目录归属规则

### 组件放哪里

- 页面私有组件：放当前页面目录附近
- 业务域内复用组件：放 `features/<domain>/components`
- 跨业务域复用组件：放 `components/common`

例子：

- 登录页里的密码输入块，只给登录页用
  放 `features/auth/screens` 邻近目录
- 登录和注册共用的验证码输入组件
  放 `features/auth/components`
- `设置 / 关于我们 / 用户信息` 共用的分组信息卡片
  放 `features/profile/components`

### API 放哪里

- 全局 API 工厂：放 `data/api.ts`
- 业务域接口：放 `features/<domain>/api.ts`
- 跨业务域复用的 schema：放 `data/schemas.ts`
- session/token：放 `data/session.ts`

例子：

- 登录、注册、找回密码
  放 `features/auth/api.ts`
- 首页卡片、首页内容
  放 `features/home/api.ts`
- 获取个人资料、更新用户信息、读取设置、主题语言配置
  放 `features/profile/api.ts`

## 推荐初始化页面

首版模版建议内置这 9 个基础页面。

### 1. `LaunchScreen`

职责：

- 读取本地 session
- 模拟启动延迟
- 决定跳转 `Login` 或 `MainTabs`

### 2. `LoginScreen`

职责：

- 手机号或邮箱输入
- 密码输入
- 登录按钮
- 注册入口
- 找回密码入口

### 3. `RegisterScreen`

职责：

- 手机号或邮箱注册
- 验证码或密码确认
- 注册提交
- 返回登录入口

### 4. `ForgotPasswordScreen`

职责：

- 账号输入
- 验证身份
- 重设密码

### 5. `HomeScreen`

职责：

- 顶部欢迎区
- 统计卡片
- 快捷入口
- Banner 或推荐内容

### 6. `MyScreen`

职责：

- 用户头像、昵称、角色
- 用户信息入口
- 设置入口
- 关于我们入口

### 7. `UserInfoScreen`

职责：

- 展示用户资料
- 编辑昵称、头像、手机号、邮箱等基础信息

### 8. `SettingsScreen`

职责：

- 主题入口
- 多语言入口
- 关于我们入口
- 退出登录入口

### 9. `ThemeScreen`

职责：

- 浅色 / 深色 / 跟随系统
- 验证主题切换链路

### 10. `LanguageScreen`

职责：

- 中文 / English 语言切换
- 验证多语言状态切换

### 11. `AboutScreen`

职责：

- 应用版本
- 公司/团队介绍
- 隐私政策、用户协议入口

## 推荐 API 写法

这里直接使用框架提供的 `createAPI`，不额外设计 `services/api/*`。

```ts
// src/data/api.ts
import { createAPI, ErrorCode } from '@gaozh1024/rn-kit';
import { appConfig } from '../bootstrap/app-config';
import type { ApiEndpointConfig } from '@gaozh1024/rn-kit';

export function createAppAPI<T extends Record<string, ApiEndpointConfig<any, any>>>(endpoints: T) {
  return createAPI({
    baseURL: appConfig.apiBaseURL,
    endpoints,
    onError: (error, context) => {
      // 统一监听：401、toast、埋点、日志
    },
    parseBusinessError: data => {
      const result = data as { success?: boolean; message?: string };
      if (result?.success === false) {
        return {
          code: ErrorCode.BUSINESS,
          message: result.message || 'Business error',
        };
      }
      return null;
    },
  });
}
```

然后在业务域里再做具体接口：

```ts
// src/features/auth/api.ts
import { z } from 'zod';
import { createAppAPI } from '../../data/api';

export const authApi = createAppAPI({
  login: {
    method: 'POST',
    path: '/auth/login',
    input: z.object({
      mobile: z.string(),
      password: z.string(),
    }),
  },
  register: {
    method: 'POST',
    path: '/auth/register',
  },
  forgotPassword: {
    method: 'POST',
    path: '/auth/forgot-password',
  },
});
```

## 第一版交付建议

如果现在要正式补一个模板，建议按这个顺序落地：

1. 建 `templates/expo-starter` 目录骨架
2. 接通 `app/_layout.tsx -> RootApp -> providers -> RootNavigator`
3. 补齐 `Launch / Login / Register / ForgotPassword / Home / My / UserInfo / Settings / Theme / Language / About`
4. 补 `bootstrap/app-config.ts`、`theme.ts`
5. 补 `data/api.ts`、`session.ts` 和 `mocks/*`
6. 最后写模板 README

这个顺序能保证每一步都能运行、能验证，也不会把模板做成一个过度设计的空架子。
