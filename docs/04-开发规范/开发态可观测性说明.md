# 开发态可观测性说明

本文说明 `@gaozh1024/rn-kit` 当前提供的开发态可观测性基础设施，包括：

- Logger
- App 内日志浮层
- React 错误边界
- API 自动打点

目标是帮助开发阶段快速定位问题，而不是直接替代线上监控平台。

## 1. 总览

框架当前有两条主要接入路径：

### 方案 A：直接通过 `AppProvider`

这是推荐方式。

```tsx
import { AppProvider } from '@gaozh1024/rn-kit';

export default function App() {
  return <AppProvider>{/* app */}</AppProvider>;
}
```

开发环境下：

- `enableLogger` 默认开启
- 可启用 `AppErrorBoundary`
- `createAPI` 的 `observability` 默认可记录请求链路

### 方案 B：手动组合

如果你的工程需要自行控制 Provider 顺序，可以单独使用：

- `LoggerProvider`
- `AppErrorBoundary`
- `OverlayProvider`

但这时要注意：

- `LoggerProvider` 默认 `enabled = false`
- `OverlayProvider` 里的 logger 也不会默认主动开启

所以通常需要显式传 `enabled: true`。

## 2. Logger

### 2.1 组件内记录日志

```tsx
import { useLogger } from '@gaozh1024/rn-kit';

function LoginButton() {
  const logger = useLogger('auth');

  return null;
}
```

支持：

- `debug`
- `info`
- `warn`
- `error`
- `clear`
- `entries`

示例：

```tsx
logger.info('开始登录', { username: 'demo' });
logger.error('登录失败', { code: 401 });
```

推荐始终使用 `namespace`，便于后续筛选：

- `auth`
- `profile`
- `network`
- `payment`
- `react`

### 2.2 `AppProvider` 中控制 logger

```tsx
<AppProvider
  enableLogger
  loggerProps={{
    level: 'info',
    maxEntries: 300,
    consoleEnabled: true,
    overlayEnabled: true,
    defaultExpanded: false,
    exportEnabled: true,
  }}
>
  <App />
</AppProvider>
```

常用参数：

- `level`
- `maxEntries`
- `consoleEnabled`
- `overlayEnabled`
- `defaultExpanded`
- `exportEnabled`
- `onExport`

### 2.3 日志导出

当启用 `exportEnabled` 时，日志浮层支持导出当前筛选结果。

```tsx
loggerProps={{
  exportEnabled: true,
  onExport(payload) {
    console.log(payload.entries);
    console.log(payload.serialized);
  },
}}
```

`payload` 结构：

```ts
{
  entries: LogEntry[];
  serialized: string;
}
```

## 3. LogOverlay

App 内日志浮层主要用于移动端开发时快速查看日志。

当前支持：

- level 筛选
- namespace 筛选
- 清空
- 导出当前结果
- 浮动按钮拖动吸边
- 浮动按钮位置记忆

推荐使用场景：

- 真机调试时快速查看接口返回
- 排查页面跳转与交互链路
- 跟踪按钮点击、表单提交、状态切换

## 4. Error Boundary

框架提供：

- `AppErrorBoundary`
- `AppProvider.enableErrorBoundary`

示例：

```tsx
<AppProvider
  enableLogger
  enableErrorBoundary
  errorBoundaryProps={{
    title: '页面发生错误',
    description: '请稍后重试',
    showDetails: true,
  }}
>
  <App />
</AppProvider>
```

当 logger 与错误边界同时开启时：

- React 渲染错误会自动写入 `react` 命名空间日志
- 方便同时在控制台和 `LogOverlay` 中回看

## 5. API 自动打点

`createAPI` 支持开发态请求自动打点。

记录阶段：

- `request`
- `response`
- `error`

其中 `error` 不仅包含：

- 网络错误
- HTTP 错误
- 业务错误

也包含请求构建阶段的错误，例如：

- `getHeaders()` 异步取 token 失败
- 请求体序列化失败

### 5.1 基础用法

```tsx
import { createAPI } from '@gaozh1024/rn-kit';

const api = createAPI({
  baseURL: 'https://api.example.com',
  observability: {
    enabled: true,
    includeInput: true,
    includeResponseData: false,
  },
  endpoints: {
    getProfile: {
      method: 'GET',
      path: '/profile',
    },
  },
});
```

### 5.2 接入 logger

如果 App 已通过 `AppProvider` / `LoggerProvider` 启用 logger，则 API 请求会自动写入日志系统。

默认命名空间通常为：

- `api`

也可以手动指定 transport：

```tsx
import { createAPI, createApiLoggerTransport } from '@gaozh1024/rn-kit';

const transport = createApiLoggerTransport({
  namespace: 'network',
  includeInput: true,
  includeResponseData: true,
});
```

## 6. 推荐实践

### 6.1 namespace 统一命名

建议按业务域命名，而不是按页面文件名随意命名：

- `auth`
- `home`
- `profile`
- `settings`
- `api`
- `react`

### 6.2 不要把开发日志等同于线上监控

当前这套能力主要面向：

- 本地开发
- 联调
- 测试阶段

如果后续要升级到生产监控，建议再补：

- 服务端上报
- 持久化
- 会话关联
- 用户设备信息
- 崩溃聚合与告警

### 6.3 控制数据量

建议：

- 开发时 `maxEntries` 不要过大
- 接口响应体默认不要全部展开
- 敏感信息不要直接记录到日志

## 7. 发布前建议检查

在准备发布新版本前，建议至少验证：

```bash
pnpm --dir packages/rn-kit typecheck
pnpm --dir packages/rn-kit test
pnpm --dir packages/rn-kit build
pnpm --dir templates/expo-starter lint
```

如果这次改动涉及 observability，还建议额外手工确认：

- `LogOverlay` 能否正常展开 / 收起
- 筛选、导出是否正常
- React 渲染错误是否能被边界接住
- API `request / response / error` 是否都有日志
