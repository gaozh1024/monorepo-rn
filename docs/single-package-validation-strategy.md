# 单包验证策略

> 本文档是下一阶段架构任务 A2 的交付物
>
> 创建日期: 2026-03-18
>
> 用途: 明确当前单包形态在真实项目中的验证方法和拆包触发条件

## 背景

根据架构决策，当前采用**单包模式**（`@gaozh1024/rn-kit`），暂不推进子入口设计和拆包维护。本策略用于指导如何在单包形态下验证框架的可用性和稳定性，并明确何时考虑拆包。

## 目标

1. 验证当前包形态在真实项目中的可用性
2. 在拆包前收集足够的实际使用反馈
3. 建立客观的拆包决策依据（而非主观判断）

---

## 验证项目选择

### 候选项目标准

| 标准     | 说明                                               |
| -------- | -------------------------------------------------- |
| 代表性   | 能代表典型的 React Native 应用场景                 |
| 规模适中 | 既不能太简单（无法验证），也不能太复杂（排错困难） |
| 可维护   | 团队能持续跟进项目更新                             |
| 可公开   | 最好能开源或部分开源，便于记录和分享问题           |

### 建议的验证项目

| 项目类型            | 用途                                    | 优先级 |
| ------------------- | --------------------------------------- | ------ |
| 示例项目 (example/) | 展示框架基础用法，作为快速验证基准      | P0     |
| 中等规模实际项目    | 验证复杂场景下的可用性                  | P1     |
| 新项目试点          | 从零开始使用框架，验证 DX（开发者体验） | P1     |

## 验证维度

### 1. 接入路径验证

验证项目能否顺利接入框架：

```bash
# 安装
npm install @gaozh1024/rn-kit

# 或本地验证
pnpm yalc:publish  # 在框架目录
cd example && yalc add @gaozh1024/rn-kit
```

**检查项**:

- [ ] 安装无冲突
- [ ] 类型定义正常
- [ ] Metro 配置无需额外调整
- [ ] 能正常渲染首个页面

### 2. 功能覆盖验证

验证核心功能是否正常工作：

| 模块          | 验证内容                 | 通过标准         |
| ------------- | ------------------------ | ---------------- |
| Theme         | 切换主题、自定义颜色     | 主题切换无闪烁   |
| UI Components | 各组件渲染和交互         | 无明显样式问题   |
| Navigation    | 页面跳转、参数传递       | 导航流畅无卡顿   |
| Overlay       | Loading/Toast/Alert      | 全局弹窗正常显示 |
| Core Hooks    | useRequest/usePagination | 数据获取正常     |
| Form          | 表单验证和提交           | 验证规则生效     |

### 3. 升级路径验证

验证版本升级是否平滑：

| 场景       | 验证内容                             |
| ---------- | ------------------------------------ |
| 小版本升级 | 无 breaking change，直接升级         |
| 中版本升级 | 有 deprecation warning，提供迁移指南 |
| 大版本升级 | 有 breaking change，提供 codemod     |

### 4. 构建验证

验证生产构建是否正常：

```bash
# iOS
npx react-native run-ios --configuration Release

# Android
npx react-native run-android --variant release
```

**检查项**:

- [ ] 构建成功
- [ ] 包体积合理（需记录基线）
- [ ] 无未使用的依赖被打包

### 5. 性能验证

| 指标       | 基线 | 目标             |
| ---------- | ---- | ---------------- |
| 启动时间   | TBD  | 不劣于纯 RN 项目 |
| 内存占用   | TBD  | 无内存泄漏       |
| 包体积增加 | TBD  | < 500KB (压缩后) |

## 拆包触发条件

### 触发条件清单

当满足以下条件之一时，启动拆包评估：

| 条件         | 阈值                 | 说明                             |
| ------------ | -------------------- | -------------------------------- |
| 包体积       | 压缩后 > 1MB         | 当前基线待测量                   |
| 安装时间     | 明显变慢             | 主观感受+客观数据                |
| 版本迭代冲突 | 不同模块版本频繁冲突 | 如 navigation 和 ui 版本节奏不同 |
| 用户需求     | 明确需要按需引入     | 如 "我只想要 UI 组件"            |
| 维护成本     | 单包问题定位困难     | 难以确定是哪个子模块的问题       |
| 稳定性       | 单模块稳定后         | 某个子模块经过 3+ 版本未大改     |

### 拆包评估流程

```
条件触发
   ↓
收集数据（使用反馈、包体积、构建时间等）
   ↓
团队讨论（成本 vs 收益）
   ↓
决策记录
   ↓
    ├── 不拆包 → 记录原因，继续观察
    └── 拆包 → 制定拆包计划
```

## 当前决策

**当前状态**: 维持单包

**理由**:

1. 框架仍在快速迭代期，API 可能调整
2. 尚未收到用户对按需引入的明确需求
3. 单包维护成本目前可控
4. 需要更多真实项目使用数据

**下次评估时间**: 完成 3 个版本迭代后（或收集到 3+ 真实项目反馈）
