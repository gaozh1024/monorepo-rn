# 移动端软件开发架构说明

本文面向使用 `@gaozh1024/rn-kit` 及其配套模板的项目团队，目标是说明三件事：

1. 如何初始化一个新项目
2. 如何理解推荐的软件架构分层
3. 如何在业务项目中使用框架与配套能力包

## 1. 适用场景

当前仓库主要提供两种使用方式：

- 直接使用 `@gaozh1024/expo-starter` 创建新项目
- 在已有 Expo / React Native 项目中接入 `@gaozh1024/rn-kit`

如果你的目标是快速启动一个新的业务 App，优先推荐模板方式。  
如果你的项目已经存在，只是想引入主题、导航、组件、API 工厂等能力，优先推荐框架接入方式。

## 2. 推荐初始化方式

### 2.1 用模板创建新项目

```bash
npx create-expo-app@latest my-app --template @gaozh1024/expo-starter
cd my-app
npm install
npm run lint
npx expo start
```

模板默认已经包含：

- Expo SDK 54 / React Native 0.81 依赖基线
- `@gaozh1024/rn-kit`
- 主题系统与 `AppProvider`
- 单层根导航结构
- 登录、首页、设置、关于等基础页面
- `expo-secure-store` 持久化 storage 注入
- deep linking 脚手架
- 业务级 Provider 扩展位

适合需要快速进入业务开发的团队。

### 2.2 在已有项目中接入框架

先安装框架与原生依赖：

```bash
npm install @gaozh1024/rn-kit
npx expo install @expo/vector-icons expo-linear-gradient expo-image
npx expo install react-native-screens react-native-safe-area-context
npx expo install react-native-gesture-handler react-native-reanimated react-native-worklets
npm install react-native-svg nativewind
npm install -D tailwindcss
```

然后完成 NativeWind 配置：

- `babel.config.js` 启用 `nativewind/babel`
- `tailwind.config.js` 的 `content` 包含业务代码和 `@gaozh1024/rn-kit/dist`

这部分详细说明可参考：

- [`TailwindCSS配置指南`](/Users/gzh/Projects/framework/rn-monorepo/docs/01-入门指南/TailwindCSS配置指南.md)

最后在应用入口挂载 `AppProvider`：

```tsx
import { AppProvider } from '@gaozh1024/rn-kit';
import { RootNavigator } from './src/navigation/RootNavigator';

export default function App() {
  return (
    <AppProvider>
      <RootNavigator />
    </AppProvider>
  );
}
```

## 3. 推荐架构分层

推荐把项目分成六层，每层职责尽量单一：

### 3.1 入口层

典型文件：

- `index.js`
- `App.tsx`
- `src/root/RootApp.tsx`

职责：

- 注册 Expo 根组件
- 导入全局样式
- 初始化持久化 storage
- 装配最顶层 Providers
- 挂载根导航

入口层不要承载具体业务页面逻辑。

### 3.2 启动配置层

典型目录：

- `src/bootstrap/`

职责：

- 应用名称、环境、版本号
- `apiBaseURL`
- 主题配置
- 启动期常量
- storage 初始化

建议把“启动时就必须知道”的配置都放在这一层，不要散落到各业务目录。

### 3.3 导航层

典型目录：

- `src/navigation/`

职责：

- 定义项目自己的路由结构
- 维护 route 常量和 route params 类型
- 组织 RootStack / Tab 结构
- 配置 linking

框架提供的是导航能力和导航 Provider，不替代业务项目自己的路由定义。

### 3.4 业务域层

典型目录：

- `src/features/auth`
- `src/features/home`
- `src/features/profile`

职责：

- 页面
- 业务域内组件
- 业务域接口

推荐按业务域拆分，而不是把所有页面都平铺到一个 `screens/` 下。  
这样后续当页面、组件、接口增多时，结构更稳定。

### 3.5 数据层

典型目录：

- `src/data/`

职责：

- API 工厂封装
- session / token 管理
- 公共 schema
- mock 数据

推荐把跨业务域复用的数据基础能力放这里，而不是把所有接口都堆到一个大文件里。

### 3.6 公共能力层

典型目录：

- `src/components/common/`
- `src/providers/`
- `src/store/`
- `src/utils/`
- `src/types/`

职责：

- 跨业务域公共组件
- 全局 Provider 扩展位
- Zustand 全局状态
- 工具函数
- 公共类型

## 5. 框架在项目中的角色

`@gaozh1024/rn-kit` 不等于“整个业务模板”，它在项目里主要负责这几类基础能力：

### 5.1 UI 与主题

常用内容：

- `AppView`
- `AppText`
- `AppButton`
- `AppPressable`
- `Card`
- `AppImage`
- `createTheme`
- `ThemeProvider`
- `useTheme`

适合统一页面样式、色板、明暗主题行为。

推荐理解方式：

- `AppView` / `AppText` 是最基础的视图与文本组件
- `AppPressable` 适合做自定义点击区域
- `AppButton` 适合业务按钮
- `Card` 适合信息块、设置块、列表块
- `AppImage` 适合图片展示，底层已适配 `expo-image`

典型写法：

```tsx
<AppView flex p={4} bg="white">
  <AppText size="xl" weight="bold">
    页面标题
  </AppText>

  <Card mt={4} p={4}>
    <AppText>这里是卡片内容</AppText>
  </Card>

  <AppButton mt={4} onPress={handleSubmit}>
    保存
  </AppButton>
</AppView>
```

如果需要统一主题，建议在项目根部创建亮色/暗色主题：

```tsx
const lightTheme = createTheme({
  colors: {
    primary: '#f38b32',
  },
});
```

### 5.1.1 页面与容器组件

常用内容：

- `SafeScreen`
- `AppScreen`
- `AppScrollView`
- `AppFlatList`
- `KeyboardDismissView`
- `KeyboardInsetView`
- `Row`
- `Col`
- `Center`

推荐用法：

- 页面根容器优先用 `AppScreen`
- 纯安全区容器用 `SafeScreen`
- 长页面内容用 `AppScrollView`
- 列表页优先用 `AppFlatList`
- 输入场景需要键盘避让时用 `KeyboardInsetView`

经验规则：

- 带 `AppHeader` 的页面通常用 `AppScreen`
- 聊天输入区、评论框、底部表单栏通常需要 `KeyboardInsetView`
- 不要一开始就直接用原生 `View` / `ScrollView` 重写一套页面容器

### 5.1.2 展示组件

常用内容：

- `Card`
- `Icon`
- `AppImage`
- `GradientView`
- `AppList`
- `Progress`

适用场景：

- `Card`：设置卡片、摘要卡片、信息区块
- `Icon`：按钮图标、状态图标、功能入口
- `AppImage`：头像、Banner、内容图片
- `GradientView`：头图、渐变 Banner、强调区域
- `AppList`：列表页快速搭建
- `Progress`：进度展示、步骤条

### 5.1.3 表单组件

常用内容：

- `AppInput`
- `Checkbox`
- `CheckboxGroup`
- `Radio`
- `RadioGroup`
- `Switch`
- `Select`
- `Picker`
- `DatePicker`
- `BottomSheetModal`

适用场景：

- `AppInput`：账号、密码、搜索框、文本输入
- `Checkbox` / `Radio`：单项选择与勾选
- `Switch`：开关类设置项
- `Select`：选项较少、需要文本标签的选择场景
- `Picker`：移动端滚轮选择
- `DatePicker`：生日、时间、日期筛选
- `BottomSheetModal`：底部弹层表单、确认面板

推荐做法：

- 普通设置项用 `Switch`
- 多选标签组用 `CheckboxGroup`
- 单选条件用 `RadioGroup`
- 需要移动端原生感更强的选择体验时优先用 `Picker`

### 5.1.4 反馈组件

常用内容：

- `Toast`
- `Alert`
- `Loading`
- `Skeleton`
- `SkeletonText`
- `SkeletonAvatar`
- `useToast`
- `useAlert`
- `useLoading`

适用场景：

- `Toast`：轻提示，例如“保存成功”
- `Alert`：确认弹窗，例如“确认删除？”
- `Loading`：全局加载态
- `Skeleton*`：列表、卡片、头像骨架屏

推荐规则：

- 成功/失败的瞬时反馈用 `Toast`
- 有风险确认时用 `Alert`
- 页面首屏等待接口时优先用 `Skeleton`
- 不要一开始就手写一套骨架屏或全局弹窗

### 5.2 导航

常用内容：

- `AppProvider`
- `NavigationProvider`
- `StackNavigator`
- `TabNavigator`
- `useNavigation`

适合统一导航容器、主题联动和导航类型入口。

补充说明：

- `StackNavigator` 适合大多数页面跳转场景
- `TabNavigator` 适合底部主导航
- `DrawerNavigator` 适合抽屉页或侧边菜单
- `AppHeader` 适合统一页面头部

推荐结构：

- 根导航用 `StackNavigator`
- 主功能分区用 `TabNavigator`
- 抽屉或工具面板按需引入 `DrawerNavigator`

典型写法：

```tsx
<StackNavigator screenOptions={{ headerShown: false }}>
  <StackNavigator.Screen name="MainTabs" component={MainTabs} />
  <StackNavigator.Screen name="Settings" component={SettingsScreen} />
</StackNavigator>
```

### 5.3 API 与错误处理

常用内容：

- `createAPI`
- `ErrorCode`
- 全局错误分类
- 可观测性 hooks

适合把接口定义、请求行为、错误分类、日志打点放到统一机制里。

推荐把接口定义写成声明式配置，而不是每个页面手写一套 `fetch`：

```tsx
const userAPI = createAPI({
  baseURL: appConfig.apiBaseURL,
  endpoints: {
    getProfile: {
      method: 'GET',
      path: '/profile',
    },
  },
});
```

适用场景：

- 统一 `baseURL`
- 自动注入 token
- 统一业务错误处理
- 统一日志与可观测性

### 5.4 应用初始化与全局能力

常用内容：

- `AppProvider`
- `AppStatusBar`
- `AppFocusedStatusBar`
- `setStorageAdapter`
- `storage`

适用场景：

- 根部统一注入主题、导航、安全区、状态栏、Overlay
- 自定义 storage 实现
- 页面级状态栏覆盖

推荐做法：

- 新项目优先用 `AppProvider`
- 持久化能力统一在启动阶段注入
- 页面如果有沉浸式背景，再单独用 `AppFocusedStatusBar`

### 5.5 Overlay 与可观测性

常用内容：

- Logger 浮层
- Error Boundary
- `useToast`
- `useAlert`
- `useLoading`

适用场景：

- 开发期定位页面报错
- API 调试
- 全局反馈统一管理

模板默认已经在开发环境里把这些能力接好了，业务页面通常直接使用 hook 即可。

### 5.6 配套能力包

除了 `@gaozh1024/rn-kit`，当前还提供四个配套包：

- `@gaozh1024/photo-album-picker`
- `@gaozh1024/aliyun-speech`
- `@gaozh1024/aliyun-push`
- `@gaozh1024/hot-updater`

适用方式：

- 相册、裁剪、图片选择场景接入 `photo-album-picker`
- 语音转文字场景接入 `aliyun-speech`
- 阿里云移动推送场景接入 `aliyun-push`
- 需要 OTA 热更新能力时接入 `hot-updater`

这几个包都建议按需安装，不要在所有项目里默认塞满。

### 5.4 Overlay 与可观测性

常用内容：

- 日志浮层
- 错误边界
- Loading / Toast / Alert

适合开发期快速定位问题，也方便统一处理全局反馈。

### 5.5 持久化能力

框架本身暴露 `storage` 和 `setStorageAdapter`，业务项目可以接入：

- `expo-secure-store`
- AsyncStorage
- 其它自定义 storage 适配器

模板当前默认使用 `expo-secure-store`。

## 6. 推荐开发流程

### 6.1 项目初始化

1. 用 `@gaozh1024/expo-starter` 创建项目
2. 安装依赖
3. 运行 `npm run lint`
4. 运行 `npx expo start`
5. 修改 `app.json`、`app-config.ts` 中的应用信息

### 6.2 业务开发

1. 先在 `navigation/routes.ts` 定义新页面路由
2. 再在对应 `features/*` 下添加页面与组件
3. 需要状态时放到 `store/`
4. 需要接口时放到业务域 `api.ts`
5. 跨域复用能力再抽到 `components/common` 或 `data`

### 6.3 扩展框架能力

如果项目需要语音、相册、推送、OTA 热更新等配套包，可以按需接入：

- `@gaozh1024/aliyun-speech`
- `@gaozh1024/photo-album-picker`
- `@gaozh1024/aliyun-push`
- `@gaozh1024/hot-updater`

原则是：

- 只在业务需要时安装
- 原生依赖优先通过 `npx expo install` 安装
- 保持 Expo SDK 与原生依赖版本一致

## 7. 常见接入方式

### 7.1 接入相册能力

```bash
npm install @gaozh1024/photo-album-picker
npx expo install expo-image expo-image-manipulator expo-media-library
```

然后在导航中注册：

- `PhotoAlbumScreen`
- `PhotoCropScreen`

### 7.2 接入语音能力

```bash
npm install @gaozh1024/aliyun-speech
npx expo install expo-audio
```

然后在启动阶段初始化语音配置。

### 7.3 接入推送能力

```bash
npm install @gaozh1024/aliyun-push aliyun-react-native-push
```

如果使用 Expo config plugin，还需要在 `app.json` 中补充插件配置。

### 7.4 接入 OTA 热更新能力

```bash
npm install @gaozh1024/hot-updater @hot-updater/react-native
```

推荐接入方式：

- 入口保留 `hotUpdater.wrapApp(App)`
- 项目侧创建自己的 `HotUpdaterProvider`
- 在启动页调用 `startLaunchUpdateCheck()`
- 非强更后台下载
- 强更直接下载并重启

Android 侧还需要确认 `MainApplication.kt` 已正确接入 OTA bundle 入口；详细说明见 `@gaozh1024/hot-updater` 包文档。
