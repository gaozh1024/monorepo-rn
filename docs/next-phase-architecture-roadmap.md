# 下一阶段架构任务路线图

## 背景

P0 和 P1 已完成，仓库边界、发布边界、文档一致性和测试基建归属已经基本收口。

下一阶段不再以“清理遗留”为主，而是进入“在单包内收敛公共 API、重整模块边界、降低后续维护成本”的架构优化阶段。

本路线图用于承接瘦身后的后续任务，作为 P2 的执行清单。

当前策略说明：

- 暂不推进子入口设计
- 暂不推进拆包维护
- 先在单包内完成真实项目验证
- 等经过多个版本迭代且稳定后，再评估是否拆包

## 阶段目标

- 收缩对外暴露面，降低 breaking change 风险
- 控制当前开发和测试成本，不提前引入拆包复杂度
- 让目录结构与职责边界一致
- 拆分高复杂度文件，降低单文件维护成本
- 把测试资源投入到高风险模块，而不是继续集中在低风险工具层

## 任务列表

### A1. 建立公共 API 清单

目标：

- 明确哪些导出属于稳定公共 API
- 明确哪些导出只是内部实现细节

动作：

- 盘点 `packages/framework/src/index.ts` 当前对外导出项
- 盘点 `core`、`ui`、`navigation`、`theme`、`overlay` 的入口导出
- 标注每个导出项的级别：稳定、待观察、内部
- 输出一份 API 白名单

交付物：

- 一份公共 API 清单文档
- 一份待下线或待迁移的导出清单

验收标准：

- 团队能明确回答“哪些 API 可以承诺稳定，哪些不能”
- 后续重构不再依赖 `export *` 的隐式暴露

### A2. 建立单包验证策略

目标：

- 先验证当前包形态在真实项目中的可用性和稳定性
- 在拆包前收集足够的实际使用反馈

动作：

- 选定 1 到 2 个真实项目或示例项目作为验证对象
- 记录接入路径、升级路径和常见问题
- 对每次版本迭代记录 API 变动、兼容性问题和构建问题
- 建立“何时考虑拆包”的触发条件

交付物：

- 单包验证记录
- 拆包触发条件清单

验收标准：

- 是否拆包不再凭主观判断，而是基于真实使用数据

### A3. 清理三方库二次导出策略

目标：

- 避免把三方库版本和框架公共 API 绑死

重点关注：

- `zod`
- `@tanstack/react-query`

动作：

- 评估继续二次导出的收益和成本
- 若保留，给出明确理由和版本策略
- 若移除，给出迁移路径和版本说明

交付物：

- 三方导出策略决策记录

验收标准：

- 团队对“为什么要不要二次导出”有明确结论

### A4. 记录并冻结当前单包边界

目标：

- 在不拆包的前提下，先把单包内部边界定义清楚

动作：

- 为 `core`、`ui`、`theme`、`navigation`、`overlay` 建立内部职责说明
- 约束跨目录引用方向，减少未来再返工
- 标记不允许继续膨胀的目录和文件
- 为后续是否拆包保留决策依据

交付物：

- 单包边界说明文档
- 内部目录职责说明

验收标准：

- 在不拆包的情况下，目录边界已经足够清晰

### B1. 重整 `core` 目录职责

目标：

- 保持“目录名 = 领域职责”

动作：

- 将通用异步状态 Hook 与错误增强逻辑拆开看待
- 评估 `useAsyncState` 是否应迁移到 `core/hooks`
- 确认 `error` 目录只保留错误类型、错误增强、错误辅助能力
- 补充迁移说明和兼容层

交付物：

- 调整后的 `core` 目录职责说明
- 必要的迁移 PR

验收标准：

- 新成员仅通过目录名就能大致理解模块职责

### B2. 为 `overlay` 建立子域边界

目标：

- 把当前聚合在单文件里的 loading / toast / alert / provider / hooks 分开

动作：

- 设计 `overlay` 子目录结构
- 将 `OverlayHost.tsx` 拆分为多个文件
- 提取公共类型和上下文定义
- 保留现有对外 API，不做无谓 breaking change

建议目录：

- `overlay/loading/*`
- `overlay/toast/*`
- `overlay/alert/*`
- `overlay/provider.tsx`
- `overlay/hooks.ts`

交付物：

- 拆分后的 `overlay` 目录结构
- 对外 API 不变的实现重构

验收标准：

- 不再存在承载多个子系统的大型 `overlay` 单文件

### B3. 拆分导航类型和 Hook 文件

目标：

- 降低导航层的认知负担

动作：

- 将 `navigation/types.ts` 拆分为：
  - 路由参数类型
  - screen options 类型
  - navigator props 类型
  - linking 类型
- 将 `useNavigation.ts` 中不同导航类型的 Hook 进行逻辑分组
- 检查是否有仅为导出方便而拼接在一起的类型定义

交付物：

- 拆分后的导航类型目录
- 更清晰的导航 Hook 结构

验收标准：

- 导航类型和 Hook 可以按目的快速定位

### C1. 重新分配测试投入

目标：

- 将测试资源优先投入到高复杂度、高装配风险区域

优先补测模块：

- `overlay`
- `navigation`
- `AppProvider`

动作：

- 为 `OverlayProvider` 补充行为测试
- 为 `useLoading` / `useToast` / `useAlert` 补充 Hook 使用约束测试
- 为 `StackNavigator` / `TabNavigator` / `DrawerNavigator` 补充关键装配测试
- 为 `AppProvider` 补充集成测试

验收标准：

- 高风险模块至少具备基础行为回归保护
- 测试结构与代码复杂度更匹配

### C2. 降低测试环境的维护成本

目标：

- 减少自定义 mock 基建的复杂度和脆弱性

动作：

- 评估现有测试 setup 是否可以拆分
- 将通用 mock、导航 mock、RN 组件 mock 分层
- 记录哪些 mock 是必须的，哪些是历史兼容产物
- 持续消除 `act(...)` 和 `react-test-renderer` 相关告警

交付物：

- 更清晰的测试基础设施结构
- 一份测试基建说明文档

验收标准：

- 测试 setup 的改动不再牵一发而动全身

## 建议执行顺序

1. A1 建立公共 API 清单
2. A2 建立单包验证策略
3. A3 决定三方库二次导出策略
4. A4 记录并冻结当前单包边界
5. B1 重整 `core` 目录职责
6. B2 拆分 `overlay`
7. B3 拆分导航类型和 Hook
8. C1 重分配测试投入
9. C2 降低测试基建复杂度

## 每项任务的统一验证要求

- `pnpm -r typecheck`
- `pnpm -r test`
- `pnpm --filter @gaozh1024/rn-kit build`
- 关键公共 API 变更需补充文档说明

## 备注

下一阶段任务应避免一次性大重构。建议每个任务单独成 PR，优先保持外部 API 稳定，只在确有收益时引入迁移成本。
